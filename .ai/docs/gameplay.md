# Code-Word — геймплей и взаимодействие (дополнение к description.md)

Документ фиксирует игровой режим, поток партии и интерфейс. Дополняет `description.md`, не заменяет его. Решения здесь приоритетнее, если где-то расходятся с черновыми формулировками исходного описания (в частности, исходная фраза «игра на двоих» — это баг описания; верный режим — 4 игрока).

## Игровой режим

**4 игрока, 2 на 2, классический Codenames.** Две команды (red / blue), в каждой ровно две роли:

- **Спаймастер (spymaster)** — видит карту ключей (цвета всех 25 карт), даёт подсказки.
- **Оперативник (operative)** — НЕ видит цвета закрытых карт, угадывает.

Итого 4 уникальных занятых слота на старте: RED-спаймастер, RED-оперативник, BLUE-спаймастер, BLUE-оперативник.

**Про расширение «4+».** Единственное место, где допустимо больше людей, — несколько оперативников в одной команде (спаймастер всегда один). Модель `players: {uid}: {role, team, ...}` это уже поддерживает без изменений. В MVP жёсткий лимит 1+1 на команду; «+» — чисто послабление лимита в UI лобби позже, бэкенд не трогается.

## Подключение игроков

Не матчмейкинг и не друзья-лист, а **шеринг короткого кода комнаты**. Один игрок создаёт комнату, передаёт код остальным троим любым способом (мессенджер, голос, рядом). Остальные входят по коду.

**Код комнаты = ID документа** `/rooms/{code}` в Firestore. Единая логика для обоих режимов кода (ниже). Вход по коду = подписка (realtime-listener) на документ с этим id.

### Два режима задания кода при создании

1. **Автогенерация** — 6 символов случайно. При записи проверка коллизии; если id занят — генерируем заново и повторяем.
2. **Своё читаемое имя** (напр. `kitchen-party`):
    - Нормализация: нижний регистр, обрезка пробелов, разрешённый набор символов (латиница, цифры, дефис), ограничение длины.
    - Проверка существования перед созданием.
    - **При коллизии — ошибка «такое имя уже занято, введите другое».** Без предложения вариантов, без автоподбора. Просто просим ввести другое имя.
    - Подтверждение перед созданием.

**Создание — через транзакцию Firestore** (проверка существования + запись атомарно), чтобы между проверкой и записью не вклинился другой клиент. Касается обоих режимов.

### Host

В комнате заводится поле `hostUid` — это создатель. Host отвечает за единоразовую генерацию борда при старте (см. «Старт игры»), чтобы не было гонки из четырёх клиентов.

## Экран: Лобби

Самый «живой» экран — четверо разбирают роли в реальном времени.

Состав:

- **Код комнаты** крупно сверху + кнопка «копировать / поделиться».
- **Две колонки RED и BLUE**, в каждой два слота: Спаймастер и Оперативник.
- Игрок видит себя и остальных; занятые/свободные слоты обновляются у всех мгновенно через realtime-listener.
- Имена из профиля (у гостя — введённое или автогенерированное).
- **Индикатор `connected`** (точка «в сети») у каждого игрока — видно, не отвалился ли кто.
- **Кнопка «Готов»** у каждого.

### Распределение ролей — два режима

1. **Вручную** — тап по свободному слоту занимает его.
2. **Случайно** — кнопка «Перемешать роли» (доступна host'у). Раскидывает 4 присутствующих игроков по 4 слотам случайно.
    - Доступна только когда в комнате уже 4 человека.
    - После рандома ручная правка слотов остаётся доступной до старта (игрок может перекинуть себя, если роль не нравится).

### Старт игры

**Автостарт при 4/4 готовых.** Условие: все 4 слота заняты И все 4 нажали «Готов».

В этот момент:
- Статус комнаты `waiting → playing`.
- **Host** (и только host) вызывает `generateBoard`, пишет борд в комнату. Один ответственный клиент — чтобы борд сгенерился один раз, без гонки.
- Остальные по realtime-подписке автоматически проваливаются на игровой экран.

### Стартовая команда

Определяется раскладкой: команда с 9 картами (против 8) ходит первой. `startingTeam` выбирается **случайно при генерации борда**. Игроки на это не влияют.

## Экран: Игра

Один борд 5×5, но **рендерится по-разному в зависимости от роли смотрящего**.

### Что видит кто (и защита от чита)

- **Спаймастер** видит цветную карту ключей: на каждой из 25 карт подсвечен её настоящий цвет (9 / 8 / 7 нейтралов / 1 убийца).
- **Оперативник** видит «слепой» борд: все закрытые карты одинаковы; цвет проявляется только когда карта **открыта**.

**Защита от чита делается по-настоящему — оперативник не получает цвета закрытых карт вообще.** Цвета не должны лежать в открытом, читаемом всеми месте.

Реализация (раздельные документы):

- **Публичная часть** борда в `/rooms/{roomId}`: слова + флаг `revealed` (+ цвет уже открытых карт). Без цветов закрытых карт.
- **Секретная карта ключей** (цвета всех карт) — в подколлекции `/rooms/{roomId}/secret/key`. Firestore-правила дают к ней доступ **только спаймастерам** этой комнаты (правило читает `players.{uid}.role` из документа комнаты через `get()`).
- При раскрытии карты (`applyGuess`) её цвет становится публичным — пишется в публичную часть. То есть `applyGuess` пишет в два места.

Это осознанное усложнение MVP: правила безопасности Firestore становятся нетривиальными (проверка роли запрашивающего uid). Детали правил — на этапе 4 (Firebase-комнаты), как помечено в исходном описании.

### Блокировка интерфейса от мискликов

Активен интерфейс только у того, чей сейчас ход. Остальным — режим просмотра.

- **Фаза clue:** поле ввода подсказки активно только у активного спаймастера. У остальных — плашка «[Команда] спаймастер думает…», борд только на просмотр.
- **Фаза guess:** карточки кликабельны только у активного оперативника. У остальных тапы по карточкам игнорируются. Спаймастеры в фазу guess просто наблюдают.

UI-уровень: `enabled = (myUid == activePlayer)`. **Дублируется правилами Firestore** (нельзя записать ход, если ты не активный игрок) — иначе UI-блок обходится.

### Поток одного хода

1. **Фаза clue.** Активный спаймастер вводит подсказку: слово + число `count` (напр. `МЕТАЛЛ 2`). Остальные ждут.
2. Подсказка пишется в `turn.clue`, фаза → `guess`. Подсказку теперь **видят все** — она не секретна.
3. **Фаза guess.** Активный оперативник тапает карты. **Нет кнопки «Закончить ход».** `count` — это **максимум тапов**, а не обязательное «ровно столько». Ход завершается либо по исчерпанию `count` тапов, либо по первой ошибке.

Реакция на тап (`applyGuess`):

| Тапнул | Результат | Ход |
| :-- | :-- | :-- |
| свою карту | засчитана, счёт команды −1 | продолжает, пока не израсходует `count` |
| нейтрал | карта открыта нейтральной | **немедленно** переходит сопернику (оставшиеся тапы сгорают) |
| чужую карту | открывается сопернику в плюс (его счёт −1) | **немедленно** переходит сопернику |
| убийцу | **мгновенный проигрыш** команды | конец игры |

**Бонусного +1 тапа нет.** Угадал все `count` верно — ход всё равно переходит сопернику (тапы кончились). Отклонение от оригинала: в классике был бонусный тап и право остановиться раньше — здесь и того, и другого нет. Проще и предсказуемее.

4. Ход переходит к другой команде, цикл с фазы clue повторяется.

### Постоянно на экране игры

- сетка 5×5 с анимацией переворота карты (это и есть «красота» вехи 2);
- индикатор чьего хода и какой фазы;
- текущая подсказка (слово + счётчик оставшихся тапов);
- счёт `redLeft` / `blueLeft` (сколько своих карт осталось открыть каждой команде);
- кто ты в этой партии (роль + команда).

## Экран: Итог

- Победитель и причина (открыли все свои карты / соперник попал в убийцу).
- Обновление статистики.
- «Сыграть снова» (новый борд в той же комнате с теми же 4 игроками) / «Выйти».

## Влияние на модель данных (дельта к description.md)

Дополнения к `/rooms/{roomId}`:

- `hostUid: string` — создатель, ответственный за генерацию борда.
- Борд в публичной части — **без цветов закрытых карт**: `{ id, word, revealed, color? }`, где `color` присутствует только у открытых (`revealed: true`) карт.
- Подколлекция `/rooms/{roomId}/secret/key` — полная карта цветов всех 25 карт; доступ по правилам только спаймастерам комнаты.
- `turn` дополнительно отражает прогресс текущей фазы guess: израсходованные тапы относительно `count` (для авто-перехода хода). Поле вида `turn.guessesMade: int` или эквивалент.

Прочее (`status`, `turn.current/phase/clue`, `players.{uid}`, `score`, `winner`, `startingTeam`, `schemaVersion`) — как в исходном описании.

## Влияние на игровой движок (чистый Kotlin)

Уточнения к сигнатурам из description.md:

- `generateBoard(words, startingTeam)` — `startingTeam` назначается случайно (команда с 9 картами ходит первой). 9 / 8 / 7 нейтралов / 1 убийца, перемешать.
- `applyGuess(card)` — реализует таблицу реакций выше: своя → продолжить (учёт `count` как максимума), нейтрал/чужой → немедленный переход хода, убийца → проигрыш. Возвращает результат, включающий: открытый цвет, новый счёт, флаг смены хода, победителя если есть.
- `submitClue(word, count)` — фаза clue → guess, фиксирует `count` как лимит тапов.
- `checkWinner()` — победа при `redLeft == 0` или `blueLeft == 0`, либо мгновенная при вскрытии убийцы.

Логика «count как максимум + прерывания» — чистая и полностью покрывается юнит-тестами без сети (веха 1).
