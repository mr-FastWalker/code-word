# Code-Word — описание проекта

## Что это

Мобильная игра для Android — интерпретация настольной игры Codenames. Онлайн-игра на двоих, правила те же, что в оригинальной настолке. Акцент на красивом интерфейсе с карточками. Рабочее название — **Code-Word** (оригинальное «Codenames» — торговая марка Czech Games Edition, поэтому используем своё название и оформление; игровая механика не патентуется).

Текущая цель — **MVP**, но с архитектурой, заложенной под расширение: мультиязычность, сменные наборы карточек, разные скины.

## Контекст разработчика

Полный новичок в Android/Kotlin-разработке и в Firebase. Имеет опыт фронтенд-разработки. Документ и дальнейшие шаги рассчитаны на пошаговое объяснение «с нуля».

## Технический стек

- **Язык:** Kotlin
- **UI:** Jetpack Compose \+ Material 3
- **Навигация:** navigation-compose
- **IDE:** Android Studio (бесплатная, официальная)
- **Сборка:** Gradle (зависимости через `build.gradle.kts` модуля `app`, version catalog в `gradle/libs.versions.toml`)
- **Бэкенд:** Firebase (без собственного сервера)
    - **Firestore** — облачная база: состояние игры, комнаты, профили, статистика. Синхронизация двух игроков через realtime-подписки (listener / Kotlin Flow).
    - **Authentication** — анонимный вход (гость) \+ Google-вход (постоянный аккаунт).
- **Локальное хранилище:** DataStore (статистика гостя на устройстве)
- **Google-вход:** play-services-auth (для MVP классический; Credential Manager — опционально позже)
- **Асинхронность:** kotlinx-coroutines (core \+ android)
- **Тесты:** JUnit (игровой движок)

## Архитектура — ключевые принципы

- **Чистый игровой движок** (pure Kotlin, без Firebase) выделен отдельно и покрыт юнит-тестами. Раскладка карт, ходы, проверка победы — всё тестируется без сети.
- **Параметризация вместо хардкода:** язык, набор слов и скин — это параметры (три справочника \+ ссылки на них в состоянии комнаты), а не зашитые константы.
- **uid есть всегда:** и у гостя (Anonymous Auth), и у залогиненного. Авторизация \= привязка credentials к существующему анонимному uid через `linkWithCredential`, без потери данных. Весь код работает с uid одинаково.
- **Правила безопасности Firestore обязательны** перед публикацией — это единственный серверный рубеж защиты, раз приложение пишет в базу напрямую.

## Расширяемость — что заложено

**Мультиязычность — два независимых уровня (не путать):**

1. **UI приложения** (кнопки, тексты) — стандартный `strings.xml` \+ `values-ru/`, `values-en/`.
2. **Контент игры** (слова на карточках) — через `wordPackId`. Язык карточек выбирается отдельно от языка интерфейса.

**Наборы карточек:** один `locale` может иметь несколько наборов (базовый, тематический, 18+) через разные `wordPackId`.

**Скины:** `skinId` в комнате — только идентификатор; сами скины (`Skin`: цвета команд, рубашка карты, форма, шрифт, текстура) лежат на клиенте в реестре `Map<String, Skin>`. Добавление скина \= дописать запись в Map, бэкенд не трогается. В MVP — один скин `classic`, но слой выбора и провайдер закладываются сразу.

## Гость vs авторизованный пользователь

|  | Гость (anonymous) | Авторизованный |
| :---- | :---- | :---- |
| uid | есть, временный | есть, постоянный |
| Может играть | да | да |
| Статистика в сессии | да | да |
| Статистика переживёт переустановку/смену устройства | нет | да |
| Профиль (имя, аватар) | локально | в облаке |

**Решения по статистике:**

- Гость — статистика копится **локально** (DataStore), в Firestore не пишется. Это не плодит мусорные `/users` от анонимов.
- `/users/{uid}` в Firestore заводится **только в момент авторизации**.
- При логине — локальная статистика один раз переносится (merge) в облако.
- Гостю показываем статистику с мягкой пометкой «несохранённая» \+ предложение войти. Без давления.

**Подсчёт статистики:** для MVP — на клиенте после игры. Если позже понадобится защита от накрутки — вынести в Cloud Function по событию `status: finished` (закладывать сейчас не нужно, путь известен).

## Экраны MVP

1. **Старт** — Создать игру / Войти по коду. Выбор языка карточек, набора, скина. Ненавязчивая кнопка «Войти» в углу.
2. **Лобби** — код комнаты (6 символов), 2 игрока, распределение команд/ролей, «Готов».
3. **Игра** — сетка 5×5, индикатор хода и фазы, ввод подсказки (spymaster), угадывание (operative), счёт. Карта ключей видна только спаймастерам.
4. **Итог** — победитель, обновление статистики, «Сыграть снова» / «Выйти».
5. **Профиль/Статистика** — показ stats; для гостя баннер «Войдите, чтобы сохранить статистику».

## Игровой движок (чистый Kotlin)

- `generateBoard(words, startingTeam): List<Card>` — выбрать 25, назначить 9/8/7-нейтр/1-убийца, перемешать.
- `applyGuess(card): GameResult` — раскрыть, пересчитать счёт, проверить убийцу/победу, передать ход.
- `submitClue(word, count)` — сменить фазу.
- `checkWinner(): Team?`

## Внешние ресурсы и аккаунты

| Ресурс | Зачем | Стоимость | Когда |
| :---- | :---- | :---- | :---- |
| Google-аккаунт | базовый для всего | бесплатно | сразу |
| Firebase Console | Auth \+ Firestore | Spark (бесплатный), карта не нужна на старте | этап Firebase |
| Google Cloud Console | OAuth для Google-входа | бесплатно в рамках нужного | при настройке Google-логина |
| Google Play Console | публикация | разовый платёж (\~25 USD — проверить актуальное) | перед публикацией |

Цифры и условия (лимиты Firebase Spark, сумма Play Console, требования к новым аккаунтам разработчиков) Google периодически меняет — проверять актуальное перед стартом каждого этапа.

Нюанс: Google-вход (OAuth) иногда требует включить биллинг (Blaze) из\-за связки с Google Cloud, но Anonymous Auth и Firestore работают на Spark.

## Этапы публикации (кратко)

- **Формат:** AAB (Android App Bundle), не APK.
- **Keystore:** генерируется один раз; потеря \= невозможность выпускать обновления. Хранить с бэкапом.
- **Play App Signing:** рекомендуется включить (Google управляет финальным ключом).
- **Перед публикацией нужны:** иконка, скриншоты, описание, возрастной рейтинг, политика конфиденциальности (URL — обязательна из\-за аккаунтов и статистики; можно хостить на GitHub Pages), анкета Data Safety.
- **Порядок выката:** internal testing → production → ревью (от нескольких дней до пары недель).

## Важные константы и решения, зафиксированные заранее

- **package name** выбрать заранее и не менять (пример: `com.codeword.app`) — после публикации не сменить.
- `google-services.json` и keystore — **не коммитить в публичный git** (`.gitignore`).
- Коды комнат — 6 символов, генерация на клиенте с проверкой коллизий.
- `schemaVersion` в документе комнаты — для будущих миграций.
- Все тексты UI — через ресурсы, ноль строк в коде.
- Слова — через `WordPackProvider`, не константой в движке.
- Оформление — через `Skin`, не хардкод в Composable.

## Порядок реализации (вехи)

1. **Движок \+ модель** — чистая логика и юнит-тесты. Без UI и сети.
2. **UI игры на локальных данных** — сетка, карточки, анимация переворота, ход — на захардкоженной комнате, без Firebase. Здесь делается «красота».
3. **Auth-слой** — `ensureSignedIn()` \+ анонимный вход (до Firestore-комнат, т.к. комнаты требуют uid).
4. **Firebase-комнаты и синхронизация** — RoomRepository, создание/вход, realtime-слушатель.
5. **Лобби и роли** — код комнаты, команды, видимость ключей только спаймастеру.
6. **Статистика** — `StatsRepository` \+ локальный store, запись результата игры.
7. **Линковка аккаунта \+ экран профиля** — Google-вход, перенос статистики, баннеры.
8. **Полировка** — мультиязычность UI, выбор набора слов, экран итога, дисконнекты.
9. **Публикация** — Play Console, подписание, карточка магазина, тестовый трек → production.

Статистика и линковка — в конце сознательно: не блокируют играбельность, удобнее делать на работающем игровом цикле. Аккаунты и расходы — максимально ближе к концу; почти весь путь проходится локально и бесплатно.

## Размер приложения и контроль веса
Ожидаемый размер скачивания для пользователя — ~15–25 МБ при аккуратном подходе. Это норма для Compose + Firebase приложения, не «раздуто».
Из чего складывается вес: основной вклад — Jetpack Compose (UI-движок тянется целиком, десяток-другой МБ — разумная плата за «красоту»). Firebase добавляет ощутимо и зависит от числа модулей — берём только firebase-auth и firebase-firestore, не подключать analytics/crashlytics/messaging/ml «на всякий случай». play-services-auth (Google-вход) — заметная добавка из Google Play Services; легче альтернатива Credential Manager (можно заменить позже). Coroutines, DataStore, Navigation — единицы МБ суммарно. Логика игры и список слов — практически ноль. Реальный риск веса — только скины с тяжёлыми текстурами.
Два механизма срезают вес почти бесплатно — нужно просто их не отключать:

R8 (минификация + shrinking) — выкидывает неиспользуемый код из библиотек, часто срезает вес в разы. Включается в release-конфигурации: isMinifyEnabled = true и isShrinkResources = true. Важно: debug-сборка минификацию не применяет и весит сильно больше — реальный размер смотреть только на release.
AAB + Play App Signing (уже выбрано) — Google генерирует под каждое устройство урезанный вариант (своя плотность экрана, архитектура, язык). Поэтому размер скачивания меньше размера сборки.

Чего избегать, чтобы не раздуть бандл: лишние Firebase/Play-services модули «на будущее»; большие PNG вместо WebP/векторов для фонов и текстур; иконки и простые формы делать через VectorDrawable (весят байты); тяжёлые библиотеки ради мелочи (целый Lottie ради одной анимации и т.п.); забыть включить R8 в release; звук/музыка в высоком битрейте.
Практические правила для проекта:

В release сразу включить isMinifyEnabled и isShrinkResources.
Firebase — только auth + firestore через BoM, без лишних модулей.
Скины и иконки — векторы где можно, WebP где нужны растры; следить за размером текстур.
Размер проверять на release-AAB через APK Analyzer в Android Studio, а не на debug-сборке.
Не добавлять зависимости «впрок» — каждую новую библиотеку оценивать на вес перед подключением.
